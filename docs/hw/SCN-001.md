## 2. 과온 알람 시나리오 (타임라인)

아래는 SP = 30°C 조건에서 과온이 발생하는 전형적인 사례이다.

```
T+0s    START 명령 → FW: STATE = HEATING, PID 히터 가동 (공격적 가열), 팬 OFF
        EC: [Run] Elapsed 카운트 시작

T+45s   온도 30.0°C 도달 → FW: EVENT:TARGET_REACHED 발송
        EC: [Run] Reach = 45.0s 표시

T+52s   온도 31.0°C 초과 (SP+1°C) → FW: warn 타이머 시작
        EC: 온도 상승 추이 관찰 중 (화면 변화 없음)

T+57s   31°C 초과 5초 지속 → FW: STATE = WARNING
        FW: EVENT:WARN,ALM-01 발송
        FW: 냉각팬 ON (강제 냉각 시작)
        FW: 부저 간헐 시작 (1s ON / 1s OFF)
        EC: [State] WARNING (황색), [Alarm] ALM-01 TEMP_WARNING 표시

        ※ PID 출력: 온도 > SP → PID 오차 음수 → 출력 0%로 클램프 → 히터 OFF
        ※ 냉각팬이 온도를 SP+1°C 이하로 낮추면 팬 OFF → WARNING 자동 해제 → HEATING 복귀

T+67s   온도 32.0°C 초과 (SP+2°C) → FW: alarm 타이머 시작
        ※ 팬 냉각에도 불구하고 온도가 계속 상승 중 → 원인 규명 필요

T+77s   32°C 초과 10초 지속 → FW: STATE = ALARM
        FW: 히터 PWM = 0% (히터 즉시 OFF)
        FW: 팬 ON 유지 (WARNING에서 이미 ON 상태)
        FW: 부저 연속 ON
        FW: EVENT:ALARM,ALM-02 발송
        EC: [State] ALARM (적색), [Alarm] ALM-02 TEMP_ALARM 표시
        EC: [복구 조건] Temperature ≤ 28.0°C : ✗ 표시

T+77s~  CS 엔지니어 현장 도착 및 진단 시작
        EC 화면 → [Alarm] 원인 확인, [KPI] 오버슈트 확인
        원인 분류 → 3절 진단 트리 참조

T+??    복구 조건 충족 (온도 ≤ 28.0°C)
        EC: [복구 조건] : ✓ 표시 → RESET 명령 입력 가능

T+??    EC CLI: RESET → FW: ACK:RESET → STATE = IDLE
        EC CLI: START → 재가동 검증
```

## 3. FW 자동 대응 시퀀스 상세

과온 발생 시 FW가 자동으로 수행하는 동작이다. CS 엔지니어 개입 전에 이미 완료된 상태이다.

### 3.1 WARNING 진입 시 (ALM-01)

| 동작 | 상세 |
|------|------|
| STATE 전환 | HEATING → WARNING |
| 히터 | PID 출력 0% (온도 > SP → PID 오차 음수 → 클램프) — 사실상 OFF |
| 팬 | ON (강제 냉각 시작) |
| 부저 | 간헐 ON (0.5s 주기) |
| UART | `EVENT:WARN,ALM-01` 발송 |
| 자동 해제 조건 | 온도 ≤ SP+1°C 복귀 시 팬 OFF, `EVENT:CLEAR,ALM-01`, HEATING 복귀 |

> **현장 판단 포인트:** WARNING 단계에서 냉각팬이 즉시 켜진다. 팬 냉각으로 온도가 SP+1°C 이하로 복귀하면 팬이 꺼지고 시스템은 HEATING으로 돌아간다. 팬 동작에도 온도가 계속 오르면 환경 온도 상승(원인 C)이거나 팬 자체 불량(원인 D)이므로 즉시 원인 확인을 시작해야 한다.

### 3.2 ALARM 진입 시 (ALM-02)

| 동작 | 상세 |
|------|------|
| STATE 전환 | WARNING → ALARM |
| 히터 | PWM = 0% (즉시 OFF) |
| 팬 | ON (강제 냉각 시작) |
| 부저 | 연속 ON |
| UART | `EVENT:ALARM,ALM-02` 발송 |
| RESET 조건 | 온도 ≤ SP - 2°C (예: SP=30°C → 28°C 이하) |

> **안전 설계 원칙:** 온도가 복구 조건을 충족하지 않으면 RESET 명령을 전송해도 `NACK:RESET,TEMP_HIGH`로 거부된다. 충분히 냉각되기 전에는 재가동이 불가능하다.

## 4. EC 진단 패널 판독

ALARM 발생 직후 EC 화면에서 다음 항목을 순서대로 확인한다.

```
[State]       FW: ALARM                          ← 현재 상태
[Sensor]      Temperature: 32.5°C   Setpoint: 30.0°C   Duty: 0.0%
                                                  ↑ ALARM 이후 히터 OFF 확인
[Run]         Elapsed: 77.3s   Reach: 45.2s   Settle: ---
                                ↑ 목표 도달 후 불안정 → 제어/열적 문제
[KPI]         Peak: 32.8°C   Overshoot: 2.8°C
                              ↑ 오버슈트 크기로 원인 단서 확보

[Alarm]  ALM-02 TEMP_ALARM
    발생: 온도 > SP+2°C, 10초 지속
    복구: 온도 ≤ SP-2°C 복귀 후 RESET

    [복구 조건] Temperature ≤ 28.0°C : ✗  (현재 32.5°C)
      → 냉각 대기 중...
```

**판독 체크리스트:**

| 확인 항목 | 의미 |
|---------|------|
| Duty: 0.0% | ALARM 이후 히터 정상 차단 확인 |
| Overshoot 크기 | 소폭(~1°C): 환경 온도 상승 / 대폭(>3°C): PID 또는 히터 과출력 |
| Reach 시간 | 매우 짧음: 히터 출력 과다 / 매우 김: 히터 출력 부족 |
| Settle 표시 | `---`: 한 번도 안정화되지 않음 → 제어 불안정 또는 열적 불균형 |

## 5. 원인 분류 진단 트리

```
과온 알람 발생 (ALM-01 / ALM-02)
        │
        ▼
[Step 1] 오버슈트 규모 확인 (EC KPI 패널)
        │
        ├── 소폭 오버슈트 (< 1°C) ──────► [Step 2-A] 환경/열적 원인
        │
        └── 대폭 오버슈트 (≥ 1°C) ──────► [Step 2-B] 제어/하드웨어 원인


[Step 2-A] 환경/열적 원인
        │
        └── 온도 기저값 상승 후 완만히 유지 ► 원인 C: 주변 환경 온도 상승


[Step 2-B] 제어/하드웨어 원인
        │
        ├── 가열 초반 급격한 온도 상승 ───► 원인 A: PID 게인 과다
        │
        ├── Duty% 높은 상태에서 WARNING ──► 원인 B: 히터 과출력 고착
        │
        └── 팬 미동작 확인 ─────────────► 원인 D: 팬/릴레이 회로 불량
```

## 6. 원인별 진단 방법 및 조치

### EC CSV 로그 분석

EC가 자동 생성하는 `logs/ec_sensor_YYYYMMDD_HHMMSS.csv`는 알람 발생 이전 타임라인을 재구성할 수 있는 유일한 기록이다. **현장 도착 후 가장 먼저 열어야 할 자료**이다.

**로그 파일 컬럼:**

```
pc_time, temperature_c, setpoint_c, duty_pct, fw_state, alarm,
elapsed_s, target_reached, reach_s, peak_temp_c, overshoot_c,
settled, settle_s, raw_line
```

**원인별 로그 패턴:**

| 원인 | 로그에서 보이는 패턴 |
|------|-------------------|
| **A. PID 게인 과다** | duty_pct가 SP 근처에서 100%↔0% 급격 반복 / overshoot 큰 값 기록 |
| **B. MOSFET 고착** | fw_statve=ALARM 이후에도 duty_pct=0이지만 temperature_c 계속 상승 |
| **C. 환경 온도 상승** | IDLE/HEATING 초반 temperature_c 기저값이 이전 로그 대비 높음 |
| **D. 팬 미동작** | fw_state=ALARM 구간에서 temperature_c 하강 없거나 매우 느림 |

**알람 직전 구간 분석 방법 (예시):**

```
1. CSV를 스프레드시트로 열기
2. fw_state = "WARNING" 첫 행 찾기 → 그 30행 앞부터 검토
3. duty_pct 컬럼: 상승 추이인지, 급격 스윙인지 확인
4. temperature_c 컬럼: 상승 기울기 계산
   → (T_warning - T_start) / elapsed_s
5. peak_temp_c, overshoot_c 값으로 오버슈트 규모 확인
6. settle_s: 값 없으면('') 한 번도 안정화 미달 → 제어 또는 열적 불안정
```

### 계측 도구 선택 기준

| 도구 | 사용 목적 | 이 프로젝트의 확인 대상 |
|------|---------|----------------------|
| **EC CSV 로그** | 알람 발생 전·후 타임라인 재구성, KPI 수치 확인 | 온도 추이, Duty% 변화, reach_s / overshoot / settle_s |
| **멀티미터** | DC 전압 레벨, 도통, 저항 | 전원 레일(24V/5V/3.3V), 릴레이 출력, MOSFET Drain 전압 |
| **로직분석기** | 디지털 신호 파형, 듀티, 프로토콜 타이밍 | PWM 듀티(PB0), GPIO 출력(PA0), DHT22 단선 프로토콜, UART 메시지 |

> **진단 순서:** CSV 로그로 사전 타임라인 파악 → EC 패널로 현재 상태 확인 → 멀티미터·로직분석기로 하드웨어 레이어 검증

### 원인 A — PID 게인 과다 (제어 파라미터 문제)

**발생 메커니즘:**
Kp가 크면 오차에 비례한 히터 출력이 과대하게 인가되어 목표 온도를 크게 초과한 뒤 진동하는 오버슈트가 발생한다.

**현장 확인 방법:**

| 확인 항목 | 도구 | 방법 | 판단 기준 |
|---------|------|------|---------|
| Overshoot 크기 | EC 패널 | KPI Overshoot 값 확인 | ≥ 2°C 이상이면 과다 |
| **알람 직전 Duty% 추이** | **CSV 로그** | **duty_pct 컬럼 — 알람 발생 30~60초 전 구간** | **WARNING 진입 직전 duty_pct 지속 80% 이상 → 출력 과다** |
| **온도-Duty 상관** | **CSV 로그** | **temperature_c vs duty_pct 동시 확인** | **duty_pct 낮은데도 온도 계속 상승 → 원인 C 가능성** |
| 온도 진동 패턴 | EC Sensor Log | Temperature 추이 관찰 | SP 부근에서 상하 진동 반복 |
| **PWM 파형 듀티** | **로직분석기** | **CH1: PB0(A3) — PWM 파형 캡처** | **EC Duty% 표시와 실제 파형 Duty 불일치 시 PWM 회로 이상** |
| **PWM 과도 스윙** | **로직분석기** | **CH1: PB0 — SP 근처 구간 파형** | **100%↔0% 급격 반복 시 Kp 과다 확정** |

**조치:**

```
1. EC CLI: STOP  →  현재 운전 정지
2. Kp 값을 현재의 50~70% 수준으로 낮춤
   (FW 코드 g_kp 값 수정 후 재플래시, 또는 추후 SET 명령 확장)
3. EC CLI: START  →  재가동 후 온도 추이 관찰
4. Overshoot가 1°C 이내로 줄어드는지 확인
```

**주의:** Kp를 너무 낮추면 목표 온도 도달 시간이 길어진다. Ki, Kd도 함께 조정이 필요할 수 있다.

### 원인 B — 히터 과출력 고착 (하드웨어 불량)

**발생 메커니즘:**
MOSFET 또는 PWM 회로 이상으로 히터 출력이 정상 이하로 줄어들지 않아 온도가 계속 상승한다.

**현장 확인 방법:**

| 확인 항목 | 도구 | 방법 | 판단 기준 |
|---------|------|------|---------|
| **ALARM 전환 후 Duty% 확인** | **CSV 로그** | **fw_state=ALARM 행의 duty_pct 확인** | **duty_pct > 0% 지속 시 FW가 히터를 끄지 못한 것** |
| **온도 하강 여부** | **CSV 로그** | **ALARM 이후 temperature_c 추이** | **Duty=0%인데 온도 계속 상승 → MOSFET Drain 회로 고착** |
| Gate 전압 레벨 | 멀티미터 | PB0(A3) 핀 DC 전압 측정 | ALARM 후 0V 미달 시 FW 또는 배선 이상 |
| Drain 전압 | 멀티미터 | 히터(+) 단자 DC 전압 측정 | Gate=0V인데 24V 유지 시 MOSFET 고착 |
| MOSFET 발열 | 비접촉 확인 | 손 근접 | 과도한 발열 시 소자 열화 |
| **Gate 구동 파형** | **로직분석기** | **CH1: PB0(A3) — ALARM 전환 순간 캡처** | **ALARM 명령 후 PWM이 즉시 0%로 전환되는지 확인 / 지연 발생 시 FW 버그 또는 드라이브 회로 이상** |
| **Gate–Drain 스위칭 타이밍** | **로직분석기** | **CH1: Gate(PB0), CH2: Drain — 동시 캡처** | **Gate LOW 후 Drain 전압 상승 지연 과다 시 MOSFET 열화 판단** |

**조치:**

```
1. 파워 서플라이 출력 OFF (24V 차단)
2. MOSFET 모듈 및 게이트 배선 점검
   - 220Ω 직렬 저항 단선 여부 확인
   - 10kΩ 풀다운 저항 연결 확인 (MCU 리셋 시 Gate = GND)
3. MOSFET 모듈 교체 후 재가동
4. ALARM 없이 SP 추종하는지 확인
```

### 원인 C — 주변 환경 온도 상승

**발생 메커니즘:**
실내 온도가 상승하거나 장비 주변에 열원이 추가되어 챔버 기저 온도가 올라가고, 상대적으로 작은 히터 출력으로도 SP를 초과하게 된다.

**현장 확인 방법:**

| 확인 항목 | 방법 | 판단 기준 |
|---------|------|---------|
| **IDLE 구간 기저 온도** | **CSV 로그** | **fw_state=IDLE 또는 HEATING 초반 temperature_c** | **IDLE 시 온도가 평소 대비 높으면 환경 온도 상승 확정** |
| **온도 드리프트 장기 추이** | **CSV 로그** | **장시간 로그에서 temperature_c 기저값 변화** | **수십 분에 걸친 완만한 상승 → 환경 원인** |
| 주변 온도 확인 | 별도 온도계 | 실내 온도 측정 | 평소 대비 높음 |
| 장비 주변 점검 | 육안 확인 | 인접 열원, 환기 차단 여부 | 이상 열원 유무 |

**조치:**

```
1. 장비 설치 환경 환기 개선
2. 인접 열원 제거 또는 이격 거리 확보
3. SP를 현재 환경 온도 + 여유값으로 재설정
   EC CLI: SET:<새_SP>
4. START 후 온도 안정화 확인
```

### 원인 D — 팬 또는 릴레이 회로 불량

**발생 메커니즘:**
WARNING 진입 시 팬이 자동으로 켜져 냉각을 시작해야 하는데 켜지지 않으면 냉각이 이루어지지 않아 온도가 계속 상승하여 ALARM으로 에스컬레이션된다. ALARM 진입 후에도 팬이 켜지지 않으면 복구 냉각이 이루어지지 않는다.

**현장 확인 방법:**

| 확인 항목 | 도구 | 방법 | 판단 기준 |
|---------|------|------|---------|
| **WARNING/ALARM 후 온도 하강률** | **CSV 로그** | **fw_state=WARNING·ALARM 구간 temperature_c 추이** | **팬 ON 후에도 하강 없거나 매우 느리면 팬 미동작 또는 냉각 불충분** |
| 팬 동작 유무 | 청각/촉각 | 팬 회전 소리/진동 | WARNING 직후 소리·진동 없으면 팬 미동작 |
| PA0 출력 전압 | 멀티미터 | PA0(A0) 핀 DC 전압 측정 | WARNING 진입 후 3.3V 미달 시 FW 이상 |
| 릴레이 동작 | 청각 | 릴레이 '딸깍' 소리 | 소리 없음 = 코일 미여자 |
| 릴레이 출력 도통 | 멀티미터 | NO–COM 도통 확인 | 도통 없으면 릴레이 불량 |
| 팬 자체 | 멀티미터·직결 | 24V 직접 인가 후 회전 | 불회전 시 팬 불량 |
| **PA0 출력 전환 타이밍** | **로직분석기** | **CH1: PA0(A0) — WARNING / ALARM 진입 순간 캡처** | **STATE=WARNING/ALARM 전환 후 PA0 HIGH 전환 지연이 1사이클(~1ms) 이내인지 확인** |
| **릴레이 응답 지연** | **로직분석기** | **CH1: PA0, CH2: 릴레이 COM-NO 간 전압** | **PA0 HIGH 후 릴레이 접점 도통까지 지연 측정 — 과도 지연 시 릴레이 코일 불량** |

**조치:**

```
① PA0 미출력 → FW 이상. STM32 재플래시 또는 핀 배선 점검
② 릴레이 미동작 → 릴레이 모듈 IN 단자 3.3V 확인 후 모듈 교체
③ DC-DC 출력 불량 → DC-DC 컨버터 입력 전압 24V 확인 후 컨버터 교체
④ 팬 자체 불량 → 팬 교체
```

> **참조:** IO-001.md 5.2절 냉각 팬 회로 배선도

### 보조 진단 — DHT22 프로토콜 및 UART 검증 (로직분석기)

과온 원인 A~D와 별도로, 알람 발생 직전 센서 또는 통신 이상이 의심될 때 로직분석기로 아래 항목을 추가 확인한다.

#### DHT22 단선 프로토콜 검증

| 확인 항목 | 채널 설정 | 정상 파형 기준 | 이상 판단 |
|---------|---------|-------------|---------|
| Start 펄스 | CH1: PB5(D4) | MCU LOW 18ms 이상 → DHT22 응답 80µs LOW·HIGH | 응답 없음 → 센서 미연결 또는 전원 불량 |
| 데이터 비트 타이밍 | CH1: PB5(D4) | '0' = 26~28µs HIGH, '1' = 70µs HIGH | 타이밍 벗어남 → 풀업 저항 불량 또는 배선 길이 과다 |
| 40비트 완성 여부 | CH1: PB5(D4) | 40비트 수신 후 종료 | 도중 단절 → ALM-03 SENSOR_ERROR 원인 |

> **DHT22는 µs 단위 타이밍이 맞아야 정상 읽기가 된다. 멀티미터로는 확인 불가 — 반드시 로직분석기 사용.**

#### UART 통신 검증

| 확인 항목 | 채널 설정 | 정상 기준 | 이상 판단 |
|---------|---------|---------|---------|
| FW → EC 메시지 | CH1: PA2(TX), 115200bps 8N1 디코드 | `DATA:...` 1초 주기 수신 | 메시지 깨짐 → 보율 불일치 또는 배선 불량 |
| EC → FW 명령 | CH1: PA3(RX), 115200bps 8N1 디코드 | `START`, `RESET` 등 정상 디코딩 | 명령 미수신 → RX 핀 배선 확인 |
| ACK/NACK 응답 시간 | CH1: PA2(TX) | 명령 수신 후 200ms 이내 응답 | 무응답 → FW 루프 블록 또는 UART ISR 이상 |

## 7. 복구 절차

원인 조치 완료 후 아래 절차로 복구를 진행한다.

```
Step 1. 냉각 대기
────────────────────────────────────────────────────
  EC 화면 [Run] → Temperature 수치 모니터링
  팬 자동 ON 동작 중 → 온도 하강 확인
  목표: Temperature ≤ SP - 2°C (예: 28°C 이하)

Step 2. 복구 조건 확인
────────────────────────────────────────────────────
  EC 화면 [복구 조건] : ✓ 표시 확인
  ✗ 상태에서 RESET 명령 입력 시 → NACK:RESET,TEMP_HIGH (정상 거부)

Step 3. RESET 명령 입력
────────────────────────────────────────────────────
  EC CLI: > RESET
  성공: ACK:RESET → STATE = IDLE
  실패: NACK:RESET,TEMP_HIGH → Step 1로 복귀

Step 4. 재가동 및 검증
────────────────────────────────────────────────────
  EC CLI: > START
  EC [Run] → Elapsed 카운트 시작 확인
  온도 정상 상승 및 SP 추종 확인
  알람 재발 없이 [Run] Settle 값 표시되면 정상 복귀
```

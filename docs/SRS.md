# 1. 개요

## 1.1 목적
본 문서는 "가상 공정 챔버 기반 장비 이상 진단·인터락·PM 시뮬레이터"의 소프트웨어 요구사항을 정의한다.

본 시스템은 실제 반도체 Fab의 3계층 구조를 재현하되, 락앤락 밀폐용기를 가상 공정 챔버로 활용하여 히터 가열, 온도 제어, 이상 진단, 인터락, 예방정비(PM) 시나리오를 구현한다.

- **STM32 NUCLEO-F411RE**: 챔버 하드웨어 제어 (히터 릴레이, 센서 수집, 도어 감지, 액추에이터 구동, FreeRTOS 기반 태스크 스케줄링)
- **macOS 기반 EC 소프트웨어 (C++)**: Equipment Controller — GEM 상태 머신 관리, 이상 진단, 인터락 판단, PM 추적, SECS/GEM 통신
- **macOS 기반 Host 프로그램 (Python)**: Host / MES 시뮬레이터 — 챔버 상태 감시, 원격 명령, PM 알림 표시 (curses GUI)

본 프로젝트는 반도체 CS 엔지니어 직무에서 요구되는 다음 핵심 역량을 실습 대상으로 한다.

- 공정 챔버 온도 이상 진단 (Fault Diagnosis)
- 하드웨어·소프트웨어 인터락 (Interlock)
- 예방정비 주기 관리 (Preventive Maintenance)
- Equipment State / Alarm / Event Report 관리
- SECS/GEM (HSMS + SECS-II) 통신

## 1.2 범위

### 포함 범위
- STM32 펌웨어: FreeRTOS 기반 태스크 구조, 챔버 센서 수집, 히터/릴레이/팬/부저 제어, 도어 감지, EC와의 내부 프로토콜 통신
- STM32 무결성: Watchdog 기반 자동 복구, Graceful Degradation
- EC 소프트웨어 (macOS, C++): GEM 상태 머신, 이상 진단 로직, 인터락 판단, PM 추적, 알람/이벤트 관리, HSMS+SECS-II 통신
- Host 프로그램 (macOS, Python): 챔버 상태/센서/알람/PM 표시, 원격 명령 전송, curses GUI

### 제외 범위
- 실제 반도체 공정 제어
- SEMI 공식 인증 수준의 SECS-II 전체 구현
- 다중 장비 클러스터 제어
- 생산 이력 관리 (MES/ERP 연동)
- 사용자 계정/권한 관리
- Recipe 관리
- 챔버 진공/압력 제어

## 1.3 시스템 목표
1. 락앤락 밀폐용기와 캡톤 히터를 이용한 가상 공정 챔버를 구성한다.
2. 챔버 온도 이상 진단 로직을 구현하여 이상 상태를 감지하고 보고한다.
3. 하드웨어 인터락(SW-102T 온도 차단기)과 소프트웨어 인터락(EC 판단)을 이중으로 구현한다.
4. 히터 누적 가동 시간 및 사이클 카운트 기반 PM 주기 관리를 구현한다.
5. HSMS(TCP/IP) 위에 SECS-II 메시지를 구현하여 EC↔Host 간 표준 통신을 구현한다.
6. GEM 상태 머신, 알람/이벤트 체계를 EC에서 구현한다.
7. FreeRTOS 기반 태스크 구조로 STM32 펌웨어의 실시간성과 모듈 독립성을 확보한다.
8. Watchdog 및 Graceful Degradation으로 단일 장애가 시스템 전체 정지로 이어지지 않도록 무결성을 보장한다.

## 1.4 문서 규칙

### 요구사항 키워드 (IEEE 29148-2018 준거)
- **shall**: 반드시 구현되어야 하는 요구사항 (검증 필수)
- **should**: 구현 권장 요구사항
- **may**: 선택 구현 요구사항

### 요구사항 품질 기준 (IEEE 29148-2018)
본 문서의 모든 요구사항은 다음 특성을 충족하도록 작성한다.
- **Necessary**: 시스템 목표 달성에 필요한 요구사항만 포함
- **Unambiguous**: 단 하나의 해석만 가능하도록 정량 기준 명시
- **Verifiable**: AC(수용 기준)를 통해 합격/불합격을 판정할 수 있음
- **Traceable**: FR↔NFR↔AC 추적 매트릭스(Section 14)로 연결 관계 명시

### 상태 소유권 (State Ownership)
STM32는 로우레벨 장치 상태(raw device state)만 보유한다.
**Equipment State, 인터락 판단, PM 추적의 authoritative owner는 EC이다.**
STM32의 센서 데이터, 도어 상태, 이벤트는 EC 판단의 입력으로만 사용된다.

## 3.2 운영 시나리오

### 정상 공정 시나리오
1. STM32에 전원이 인가되고 초기화가 완료된다.
2. EC 소프트웨어가 실행되어 STM32와 UART 연결을 수립한다.
3. EC는 HSMS Passive 모드로 Host 연결을 대기한다.
4. Host가 EC에 HSMS 연결을 수립하고 Online 절차를 수행한다.
5. 운영자가 도어를 닫고 Host에서 START 명령을 전송한다.
6. EC는 인터락 조건(도어 닫힘, 정상 온도) 확인 후 히터 ON 명령을 STM32에 전송한다.
7. STM32는 릴레이를 ON하여 히터를 가동하고, 팬을 구동하지 않는다.
8. SHT31이 챔버 온도 상승을 감지하고, INA219가 히터 전류를 모니터링한다.
9. EC는 주기적으로 S6F11 Event Report를 Host에 전송한다.
10. 운영자가 STOP 명령을 전송하면 EC는 히터 OFF, 팬 ON(냉각)으로 전환 후 IDLE로 복귀한다.

### 이상 진단 시나리오
1. 챔버 온도가 경고 임계값(65°C)을 초과한다.
2. EC는 ALARM 상태로 전이하고 히터 OFF, 팬 ON, 부저 ON 명령을 전송한다.
3. S5F1 알람 보고가 Host에 전송된다.
4. 운영자가 원인 제거 후 ACK_ALARM 명령을 전송하면 IDLE로 복귀한다.

### 하드웨어 인터락 시나리오
1. 소프트웨어 알람이 해제되지 않은 채 온도가 72°C에 도달한다.
2. SW-102T(온도 차단기)가 동작하여 히터 회로를 물리적으로 차단한다.
3. INA219가 히터 전류 소실을 감지하고 EC에 보고한다.
4. EC는 INTERLOCK 상태로 전이하고 S5F1으로 Host에 보고한다.
5. 운영자가 챔버를 냉각 후 SW-102T를 수동 복구하고 RESET 명령을 전송한다.

### 도어 인터락 시나리오
1. RUNNING 중 도어가 열린다.
2. 리미트 스위치가 도어 열림을 감지하여 STM32가 EC에 보고한다.
3. EC는 즉시 INTERLOCK 상태로 전이하고 히터 OFF, 부저 ON 명령을 전송한다.
4. 운영자가 도어를 닫고 RESET 명령을 전송하면 IDLE로 복귀한다.

### PM 시나리오
1. 히터 누적 가동 시간이 100시간 또는 사이클 카운트가 500회를 초과한다.
2. EC가 PM_DUE 알람(ALID-7)을 발생하고 Host에 S5F1로 보고한다.
3. 운영자가 PM 수행 후 Host에서 PM_RESET 명령을 전송하면 카운터가 초기화된다.

# 4. 용어 정의

## 4.1 Equipment State
EC가 소유하는 장비의 동작 상태 집합

| 상태 | 설명 |
|------|------|
| IDLE | 대기 상태. 히터 OFF |
| RUNNING | 가동 상태. 히터 ON, 온도 상승/유지 |
| ALARM | 소프트웨어 이상 상태 (복구 가능, 운영자 확인 필요) |
| INTERLOCK | 인터락 상태 (하드웨어/소프트웨어 인터락 발동, RESET 필요) |

## 4.2 Control State
장비 제어 주체를 나타내는 상태 집합

| 상태 | 설명 |
|------|------|
| LOCAL | STM32 로컬 버튼 입력에 의해 제어 |
| REMOTE | Host 명령(S2F41)에 의해 제어 |

## 4.3 Fault Diagnosis (이상 진단)
수집된 센서 데이터를 기반으로 챔버 또는 히터의 이상 상태를 감지하는 로직. EC에서 수행한다.

## 4.4 Interlock (인터락)
안전 또는 보호를 위해 특정 조건에서 장비 동작을 강제 차단하는 메커니즘.
- **하드웨어 인터락**: SW-102T 온도 차단기가 72°C 이상에서 히터 회로를 물리적으로 차단
- **소프트웨어 인터락**: EC가 도어 열림 또는 심각 이상 감지 시 히터 OFF 명령 전송 및 INTERLOCK 상태 전이

## 4.5 Preventive Maintenance (PM, 예방정비)
장비 고장을 예방하기 위해 사전에 수행하는 정기 유지보수. 히터 누적 가동 시간 및 ON/OFF 사이클 카운트를 기준으로 주기를 판단한다.

## 4.6 Alarm
이상 조건 발생 시 EC가 생성하는 경고 신호. S5F1로 Host에 전달.

## 4.7 Collection Event (CE)
상태 전이, 명령 수신, 알람 등 시스템 중요 동작. S6F11로 Host에 전달.

## 4.8 Status Variable (SV)
Host가 S1F3으로 조회 가능한 장비 상태 데이터 항목.

## 4.9 HSMS
High-Speed Message Services (SEMI E37). TCP/IP 기반 SECS 전송 계층 프로토콜.

## 4.10 SECS-II
SEMI Equipment Communications Standard Part 2 (SEMI E5). Stream/Function 구조 메시지 규격.

## 4.11 SW-102T
72°C에서 동작하는 바이메탈 온도 차단기. 히터 회로에 직렬 연결되어 과온 시 히터 전원을 물리적으로 차단한다.

## 4.12 FreeRTOS
STM32에 탑재되는 오픈소스 실시간 운영체제(RTOS).

## 4.13 Task Watchdog
FreeRTOS 각 태스크별 독립적인 생존 확인 메커니즘. 특정 태스크가 설정 시간 내 체크인하지 않으면 해당 태스크만 재시작한다.

# 5. 전제조건 및 제약사항

## 5.1 전제조건
- STM32 보드와 MacBook이 USB로 연결되어 있어야 한다.
- EC와 Host는 동일 MacBook에서 별도 터미널 프로세스로 동작하며, HSMS 통신은 localhost(127.0.0.1:5000)를 사용한다.
- 12V 3A 어댑터가 DC 잭을 통해 연결되어 있어야 한다.
- 캡톤 히터가 락앤락 용기 내부에 부착되고, SW-102T가 히터 회로에 직렬 연결되어 있어야 한다.
- SHT31, INA219, 리미트 스위치, 릴레이가 지정 핀에 올바르게 배선되어 있어야 한다.

## 5.2 제약사항
- STM32 펌웨어는 STM32 NUCLEO-F411RE에서 동작해야 한다.
- EC 소프트웨어 및 Host 프로그램은 macOS 환경에서 동작해야 한다.
- EC와 Host는 동일 머신에서 독립된 프로세스로 실행되며, 논리적 계층 구조는 유지된다.
- EC↔Host 통신은 HSMS (TCP/IP) + SECS-II를 사용해야 한다.
- STM32↔EC 통신은 UART / USB Virtual COM을 사용해야 한다.
- 히터 전원은 12V DC 외부 어댑터를 사용하며 STM32 GPIO는 릴레이 제어 신호(5V 이하)만 담당한다.
- 본 프로젝트는 학습/포트폴리오 목적의 부분 구현이다.

# 6. 기능 요구사항

---

## 6.1 [STM32] 챔버 센서 수집

### FR-001 온습도 수집
STM32는 SHT31 센서로부터 FR-003에 정의된 수집 주기로 챔버 내부 온도(단위: °C) 및 습도(단위: %) 데이터를 수집해야 한다.

### FR-002 히터 전류 수집
STM32는 INA219 센서로부터 FR-003에 정의된 수집 주기로 히터 전류(단위: mA) 및 전압(단위: V) 데이터를 수집해야 한다.

### FR-003 수집 주기
센서 데이터 수집 주기는 설정값으로 관리되어야 하며 기본값은 1,000ms로 한다. 설정 가능 범위는 100ms 이상 10,000ms 이하로 한다.

### FR-004 데이터 유효성 검사
STM32는 수집된 센서 데이터가 허용 범위를 벗어나거나 I2C 읽기 실패 시, 오류 유형 코드를 포함한 오류 메시지를 EC에 전송해야 한다.

| 센서 | 항목 | 허용 범위 |
|------|------|---------|
| SHT31 | 온도 | −40°C 이상 125°C 이하 |
| SHT31 | 습도 | 0% 이상 100% 이하 |
| INA219 | 전류 | 0mA 이상 3,200mA 이하 |
| INA219 | 전압 | 0V 이상 26V 이하 |

---

## 6.2 [STM32] 도어 감지

### FR-005 도어 상태 감지
STM32는 마이크로 리미트 스위치를 통해 챔버 도어의 열림/닫힘 상태를 감지해야 한다.

### FR-006 도어 이벤트 전송
도어 상태가 변경(열림↔닫힘)될 때마다 STM32는 즉시 도어 이벤트 프레임을 EC에 전송해야 한다.

---

## 6.3 [STM32] 액추에이터 제어

### FR-007 히터 릴레이 제어
EC의 명령에 따라 릴레이를 ON/OFF하여 챔버 히터(12V/10W)를 구동 또는 정지해야 한다.

### FR-008 팬 제어
EC의 명령에 따라 냉각 팬을 구동 또는 정지해야 한다. 히터 OFF 후 냉각 모드에서 사용된다.

### FR-009 부저 제어
EC의 명령에 따라 부저를 활성화 또는 비활성화해야 한다.

### FR-010 LED 상태 표시
EC로부터 수신한 Equipment State에 따라 IDLE / RUNNING / ALARM / INTERLOCK을 구분하여 LED로 표시해야 한다.

### FR-011 로컬 버튼 처리
버튼 입력 감지 시 즉시 EC에 버튼 이벤트를 전송해야 한다.

---

## 6.4 [EC] 이상 진단

### FR-012 온도 이상 진단
EC는 수신된 챔버 온도가 TEMP_HIGH 임계값(기본값: 65°C)을 초과하는 경우 온도 이상으로 진단해야 한다.

### FR-013 히터 전류 이상 진단
EC는 다음 조건에서 히터 전류 이상으로 진단해야 한다.
- 히터 ON 상태에서 히터 전류가 정상 범위(50mA~2,000mA) 이하인 경우 (히터 단선 또는 SW-102T 동작)
- 히터 OFF 상태에서 히터 전류가 10mA를 초과하는 경우 (릴레이 불량)

### FR-014 센서 오류 진단
EC는 STM32로부터 센서 오류 코드를 수신한 경우 해당 센서 이상으로 진단해야 한다.

### FR-015 SW-102T 동작 감지
EC는 히터 ON 명령 전송 후 히터 전류가 정상 범위 이하이고 챔버 온도가 70°C 이상인 경우 SW-102T 동작으로 판단해야 한다.

---

## 6.5 [EC] 인터락

### FR-016 도어 인터락
EC는 RUNNING 상태에서 도어 열림 이벤트 수신 시 즉시 INTERLOCK 상태로 전이하고 히터 OFF, 부저 ON 명령을 STM32에 전송해야 한다.

### FR-017 소프트웨어 과온 인터락
EC는 챔버 온도가 INTERLOCK 임계값(기본값: 68°C)을 초과하는 경우 INTERLOCK 상태로 전이하고 히터 OFF 명령을 전송해야 한다.

### FR-018 시작 전 인터락 검사
EC는 START 명령 수신 시 아래 조건을 모두 만족하는 경우에만 RUNNING 상태로 전이해야 한다.
- 도어 닫힘 상태
- 활성 인터락 없음
- 센서 정상 상태

### FR-019 INTERLOCK 복구
EC는 RESET 명령 수신 시 인터락 원인이 해소된 경우에만 IDLE 상태로 복구해야 한다.

---

## 6.6 [EC] 예방정비 (PM) 추적

### FR-020 히터 가동 시간 누적
EC는 히터 ON 상태의 누적 시간을 초 단위로 추적해야 한다.

### FR-021 히터 사이클 카운트
EC는 히터 ON→OFF 전환 횟수(사이클)를 누적 계수해야 한다.

### FR-022 PM 임계값 관리
PM 임계값은 설정값으로 관리되어야 하며 기본값은 아래와 같다.

| 항목 | 기본 임계값 |
|------|------------|
| 히터 누적 가동 시간 | 100시간 |
| 히터 사이클 카운트 | 500회 |

### FR-023 PM 알람 발생
EC는 히터 가동 시간 또는 사이클 카운트가 임계값을 초과하는 경우 PM_DUE 알람(ALID-7)을 발생하고 S5F1로 Host에 보고해야 한다.

### FR-024 PM 카운터 초기화
EC는 Host로부터 PM_RESET 원격 명령 수신 시 히터 가동 시간 및 사이클 카운트를 0으로 초기화해야 한다.

---

## 6.7 [STM32] EC 통신 — 기능

### FR-025 센서 데이터 전송
수집된 센서 데이터를 EC에 주기적으로 전송해야 한다.

### FR-026 도어 이벤트 전송
도어 상태 변경 시 즉시 EC에 전송해야 한다.

### FR-027 버튼 이벤트 전송
버튼 입력 시 즉시 EC에 전송해야 한다.

### FR-028 ACK 응답
STM32는 EC로부터 수신한 명령 프레임에 대해 100ms 이내에 ACK를 전송해야 한다.

---

## 6.8 [STM32] EC 통신 — 신뢰성

### FR-029 프레임 무결성 검사
CRC16-CCITT를 사용하여 수신 프레임의 무결성을 검증해야 한다.

### FR-030 시퀀스 번호 관리
중복 수신 방지를 위해 시퀀스 번호를 사용해야 한다.

### FR-031 재전송 처리
STM32는 명령 프레임 전송 후 200ms 이내 ACK 미수신 시 최대 3회까지 재전송해야 한다.

### FR-032 Heartbeat
STM32는 5,000ms 주기로 Heartbeat 프레임을 EC에 전송해야 한다. EC는 10,000ms 이상 Heartbeat 미수신 시 UART_COMM_ERROR 알람을 발생해야 한다.

---

## 6.9 [EC] GEM 상태 머신

### FR-033 Equipment State 관리
EC는 아래 상태 집합과 전이 규칙을 구현해야 한다.

```
              CMD_START
    IDLE ────────────────────► RUNNING
     ▲                            │
     │  CMD_STOP (냉각 완료 후)   │ 이상 감지
     │◄────────────────────────── │
     │                            ▼
     │  RESET (원인 해소 후)   ALARM
     │◄────────────────────────── │
     │                            │ 임계값 초과
     │  RESET (원인 해소 후)      ▼
     │◄──────────────────── INTERLOCK
```

### FR-034 Control State 관리
EC는 LOCAL / REMOTE Control State를 관리해야 한다.

### FR-035 상태 전이 S6F11 전송
EC는 Equipment State 전이 발생 시 S6F11 Collection Event Report를 Host에 전송해야 한다.

---

## 6.10 [EC] 알람 관리

### FR-036 알람 발생
EC는 이상 진단 결과 또는 인터락 조건 발생 시 해당 ALID의 알람을 활성화하고 S5F1 Alarm Report를 Host에 전송해야 한다.

### FR-037 알람 해제
알람 원인 해소 시 EC는 해당 알람을 비활성화하고 S5F1(AlarmSet=False)을 Host에 전송해야 한다.

### FR-038 알람 확인 명령
Host의 ACK_ALARM 명령(S2F41) 수신 시 해당 알람의 운영자 확인 상태를 갱신해야 한다.

---

## 6.11 [EC↔Host] SECS/GEM 통신

### FR-039 HSMS 연결 관리
EC는 HSMS Passive 모드로 동작하며 Host의 연결 요청을 수락해야 한다.

### FR-040 Online 절차
S1F13(Online Request) 수신 시 S1F14(Online Response)로 응답해야 한다.

### FR-041 센서 데이터 S6F11 전송
EC는 STM32로부터 센서 데이터를 수신할 때마다 S6F11 CEID-8(센서 데이터)을 Host에 전송해야 한다.

### FR-042 원격 명령 처리
EC는 Host의 S2F41 Remote Command를 수신하여 처리해야 한다.

| 명령 | 동작 |
|------|------|
| START | IDLE → RUNNING 전이 (인터락 검사 후) |
| STOP | RUNNING → 냉각 → IDLE 전이 |
| RESET | ALARM/INTERLOCK → IDLE 전이 (원인 해소 후) |
| ACK_ALARM \<alid\> | 지정 알람 확인 처리 |
| PM_RESET | PM 카운터 초기화 |

### FR-043 Status Variable 응답
EC는 Host의 S1F3(Selected Equipment Status Request)에 대해 S1F4로 응답해야 한다.

| SVID | 항목 |
|------|------|
| 1 | Equipment State |
| 2 | 챔버 온도 (°C) |
| 3 | 챔버 습도 (%) |
| 4 | 히터 전류 (mA) |
| 5 | 도어 상태 (0=닫힘, 1=열림) |
| 6 | 히터 누적 가동 시간 (초) |
| 7 | 히터 사이클 카운트 |

---

## 6.12 [STM32] 무결성

### FR-044 Hardware Watchdog
STM32는 IWDG(Independent Watchdog)를 활성화하고 정상 동작 중 주기적으로 갱신해야 한다.

### FR-045 Task Watchdog
TaskWatchdog은 100ms 주기로 각 태스크의 체크인 상태를 감시하고, 설정 시간 초과 시 해당 태스크를 재시작해야 한다.

### FR-046 재시작 보고
태스크 재시작 발생 시 재시작 원인 코드를 포함한 프레임을 EC에 전송해야 한다.

### FR-047 Graceful Degradation
단일 센서 또는 액추에이터 장애 시 나머지 기능은 계속 동작해야 한다.

# 7. 비기능 요구사항

### NFR-001 센서 수집 지연
STM32 센서 수집 주기 오차는 ±10ms 이하여야 한다.

### NFR-002 UART 통신 지연
STM32→EC 메시지 전송 지연은 50ms 이하여야 한다.

### NFR-003 인터락 응답 시간
EC의 인터락 판단 및 히터 OFF 명령 전송은 인터락 조건 감지 후 200ms 이내에 완료되어야 한다.

### NFR-004 HSMS 응답 시간
EC는 Host의 요청에 대해 1,000ms 이내에 응답해야 한다.

### NFR-005 메모리 제약
STM32 전체 태스크 스택 합계는 4,096 words 이하로 제한한다.

### NFR-006 코드 가독성
각 모듈은 단일 책임 원칙을 준수하고, 모듈 간 인터페이스는 헤더 파일로 명시해야 한다.

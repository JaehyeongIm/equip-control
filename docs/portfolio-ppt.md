3. 왜 이 프로젝트를 만들었는가

칠러 장비는 공정 장비의 온도를 안정적으로 유지하기 위해 사용됩니다. 목표 온도를 벗어나면 제품 품질과 장비 안정성에 영향을 줄 수 있기 때문에, CS 엔지니어는 알람 발생 원인을 빠르게 좁히고 안전하게 복구할 수 있어야 합니다.

제가 이 프로젝트에서 보여주고 싶었던 역량은 단순한 코딩 능력이 아니라, 장비 이상 상황을 다음과 같은 순서로 바라보는 태도입니다.

1) 현재 온도가 목표 범위를 벗어났는지 확인한다.
2) 알람이 어떤 조건에서 발생했는지 확인한다.
3) 냉각 장치가 실제로 동작하는지 확인한다.
4) 히터 또는 열원이 정상적으로 차단되는지 확인한다.
5) 센서값과 로그가 실제 현상과 맞는지 확인한다.
6) 온도가 충분히 안정된 뒤에만 재가동을 허용한다.

칠러 CS 현장 업무와 프로젝트 연결:
- 목표 온도 이탈 확인: 목표 온도와 현재 온도 비교
- 과온 알람 확인: WARNING / ALARM 단계 구분
- 냉각 동작 점검: 24V 팬 자동 동작 확인
- 안전 정지: ALARM 상태에서 히터 자동 차단
- 복구 판단: 온도 조건 만족 전 RESET 거부
- 원인 추적: 로그, 전원, 신호, 부품 상태 확인


4. 시스템 개념

이 테스트벤치는 실제 칠러의 냉매, 압축기, 냉각수 유량, 압력 제어를 구현한 장비가 아닙니다. 대신 칠러 CS 직무에서 중요한 온도 이상 대응 흐름을 작게 재현하기 위해 다음과 같이 단순화했습니다.

실제 칠러 관점과 테스트벤치 단순화:
- 공정 장비 또는 열 부하: 24V 카프톤 히터
- 냉각 동작: 24V 냉각팬
- 온도 피드백: DHT22 온도센서
- 장비 제어부: STM32F446RE
- 전원 구동부: MOSFET, 릴레이
- 현장 모니터링: Python PC 화면, CSV 로그

구조 요약:
온도센서가 현재 온도를 측정합니다.
STM32 제어보드가 온도 상태를 판단합니다.
정상 범위에서는 히터 출력을 제어합니다.
주의 단계에서는 냉각팬을 동작시킵니다.
위험 단계에서는 히터를 차단하고 알람을 발생시킵니다.
PC 모니터링 화면에서는 현재 상태, 알람 코드, 복구 가능 여부, 로그 저장 상태를 확인합니다.

이 구조를 통해 온도가 올라가는 상황을 만들고, 목표 범위를 벗어났을 때 알람을 발생시키며, 냉각팬과 히터 차단 동작이 실제로 수행되는지 확인했습니다.


5. 하드웨어 및 소프트웨어 구성

온도 측정:
DHT22를 사용해 챔버 내부 공기 온도 변화를 확인했습니다. 목표 온도 대비 현재 온도가 얼마나 벗어났는지 판단하는 기준으로 사용했습니다.

가열:
24V 카프톤 히터를 사용해 과온 상황을 재현했습니다. 히터가 켜질 때 온도가 상승하고, ALARM 상태에서 차단되는지 확인했습니다.

냉각:
24V 냉각팬과 릴레이를 사용했습니다. WARNING 상태에서 팬이 자동으로 켜지고, 온도가 내려가는지 확인했습니다.

제어:
STM32F446RE가 온도 상태를 판단하고 히터, 팬, 부저 동작을 제어했습니다. NORMAL, WARNING, ALARM 상태 전환을 구현했습니다.

전원 제어:
MOSFET을 사용해 히터 전원을 제어했습니다. ALARM 상태에서 히터가 차단되는지 확인했습니다.

모니터링:
Python 기반 PC 화면에서 현재 온도, 목표 온도, 상태, 알람 코드, 히터 상태, 팬 상태, 복구 가능 여부를 확인하도록 구성했습니다.

진단 도구:
멀티미터, 로직분석기, DC 파워서플라이를 사용해 전압, 제어 신호, 부품 동작 여부를 분리해서 확인했습니다.

센서 사용 한계:
DHT22는 정밀 표면온도 센서가 아니라 챔버 내부 공기 온도 변화를 보기 위한 센서로 사용했습니다. 향후에는 히터 표면온도와 냉각 응답을 더 정확히 보기 위해 DS18B20 또는 K-type 열전대 센서 추가를 검토하고 있습니다.


6. 과온 알람 및 안전 대응 로직

장비가 목표 온도를 조금 벗어난 상황과 위험 수준으로 벗어난 상황을 구분하기 위해 상태를 NORMAL, WARNING, ALARM으로 나누었습니다.

NORMAL 상태:
발생 조건은 목표 온도 범위 내에 있을 때입니다. 이 상태에서는 히터 출력을 조절하며 정상 운전 상태로 판단합니다.

WARNING 상태:
목표 온도보다 1°C 이상 초과한 상태가 5초 이상 지속되면 WARNING으로 전환합니다. 이때 냉각팬을 켜고 PC 화면에는 노란색 상태로 표시합니다. 이 단계는 온도가 더 올라가기 전에 냉각 동작을 확인하는 주의 단계입니다.

ALARM 상태:
목표 온도보다 2°C 이상 초과한 상태가 10초 이상 지속되면 ALARM으로 전환합니다. 이때 히터를 차단하고, 부저를 켜며, PC 화면에는 빨간색 상태로 표시합니다. 이 단계는 장비 보호를 위해 안전 정지가 필요한 위험 단계입니다.

RESET 조건:
RESET 명령을 입력해도 바로 재가동하지 않도록 구성했습니다. 현재 온도가 충분히 내려갔는지 먼저 확인하고, 조건을 만족할 때만 재가동을 허용했습니다. 조건을 만족하지 않으면 RESET 명령을 거부하도록 만들었습니다.

직무 연결:
칠러 CS 현장에서도 알람을 단순 해제하는 것이 아니라, 온도가 안정 범위로 돌아왔는지 확인한 뒤 재가동 여부를 판단해야 합니다. 이 프로젝트는 그 판단 흐름을 작은 하드웨어로 재현한 것입니다.


7. PC 모니터링 화면과 로그 기반 진단

현장 엔지니어가 장비 상태를 빠르게 파악할 수 있도록 PC 모니터링 화면을 구성했습니다. 화면에서는 현재 온도, 목표 온도, 상태, 히터 출력, 알람 코드, 복구 가능 여부, 로그 저장 여부를 확인할 수 있습니다.

화면에서 확인할 수 있는 정보:
- 현재 온도: 센서가 읽은 실시간 온도
- 목표 온도: 제어 기준 온도
- 상태 표시: NORMAL / WARNING / ALARM
- 히터 상태: ON / OFF
- 팬 상태: ON / OFF
- 알람 코드: 원인 분류와 조치 방향 확인
- RESET 가능 여부: 재가동 허용 또는 거부 판단
- CSV 로그: 시간별 온도와 상태 기록

알람 코드 예시:
- ALM-01: 온도 주의 단계. 냉각팬 동작을 확인해야 합니다.
- ALM-02: 과온 위험 단계. 히터 차단 여부와 냉각 상태를 확인해야 합니다.
- ALM-03: 센서 오류. 센서 배선과 통신 상태를 확인해야 합니다.

직무 연결:
칠러 CS 현장에서는 알람 코드와 로그를 보고 원인을 좁혀야 합니다. 이 화면은 상태 확인, 알람 판단, 복구 조건 확인 과정을 시각화한 결과입니다.

삽입 자료:
ALARM 상태의 PC 화면 캡처, 알람 코드가 표시된 화면, CSV 로그 또는 온도 변화 그래프


8. 고장 재현 테스트: 냉각팬 미동작

과온 알람이 발생했을 때 단순히 “팬이 고장났다”고 판단하지 않고, 신호, 전원, 부품, 센서, 로그를 순서대로 확인하는 진단 절차를 검증했습니다.

재현한 고장 상황:
STM32 제어보드와 릴레이 입력선 사이를 의도적으로 분리했습니다. 이 상태에서 WARNING 조건이 발생해도 냉각팬이 켜지지 않고, 온도가 계속 상승하면서 ALARM으로 전환되는 상황을 만들었습니다.

관찰한 증상:
- WARNING 상태로 전환되었지만 팬이 켜지지 않았습니다.
- 온도가 내려가지 않고 계속 상승했습니다.
- 이후 ALARM 상태로 전환되며 히터가 차단되었습니다.

진단 순서:
1) PC 화면에서 WARNING 상태와 팬 ON 명령 여부를 확인했습니다.
2) STM32 제어보드에서 팬 제어 신호가 출력되는지 확인했습니다.
3) 릴레이 입력단까지 제어 신호가 전달되는지 확인했습니다.
4) 팬을 DC 파워서플라이에 직접 연결해 팬 자체가 정상인지 확인했습니다.
5) 온도 로그를 확인해 팬 미동작 시 온도가 내려가지 않는 현상을 확인했습니다.
6) 제어선을 재연결한 뒤 WARNING 단계에서 팬이 정상 동작하고 온도가 하강하는지 확인했습니다.

사용 도구와 확인 내용:
- PC 모니터링 화면: 상태 전환, 팬 명령, 알람 코드 확인
- 멀티미터: 릴레이 입력단 전압 확인
- 로직분석기: 제어 신호 출력 여부 확인
- DC 파워서플라이: 팬 자체 동작 여부 확인
- CSV 로그: 온도 변화와 알람 발생 시점 확인

원인 판단:
팬 자체 고장이 아니라, 제어보드와 릴레이 사이의 제어선 단선 문제로 판단했습니다.

직무 연결:
온도가 내려가지 않을 때 부품부터 교체하는 것이 아니라, 명령, 신호, 전원, 부품, 센서, 로그를 순서대로 확인하는 진단 습관을 보여주는 사례입니다. 이는 칠러 CS 엔지니어가 현장에서 냉각 불량 원인을 좁힐 때 필요한 접근 방식과 연결됩니다.

삽입 자료:
팬 미동작 재현 사진, 릴레이 입력선 분리 사진, 멀티미터 측정 사진, 로직분석기 캡처, 온도 로그 그래프


9. 24V 회로 측정 리스크 분석과 재발 방지 절차

개발 중 24V 전원 라인과 3.3V 신호 라인이 함께 존재하는 회로를 측정하는 과정에서 INA219 전류 센서에 스파크와 연기가 발생했고, STM32와 SHT31 센서도 함께 손상되는 문제가 있었습니다.

이 경험을 단순한 실수로 끝내지 않고, 전원 라인과 신호 라인이 가까운 회로에서 발생할 수 있는 측정 리스크로 분석했습니다. 이후 같은 문제가 반복되지 않도록 측정 순서와 예방 체크리스트를 정리했습니다.

당시 상황:
- 24V 회로 전압을 측정하는 중이었습니다.
- 프로브가 INA219 주변 핀에 순간적으로 접촉했습니다.
- 이후 INA219에서 스파크와 연기가 발생했습니다.
- STM32와 SHT31 센서도 함께 손상되었습니다.

원인:
- 뾰족한 프로브가 24V 측정 지점과 인접 신호 핀을 동시에 접촉했습니다.
- 24V가 3.3V 신호 라인으로 유입되었습니다.
- 같은 신호 라인에 연결된 부품들이 함께 손상되었습니다. (STM32, INA219, SHT31 손상)

재발 방지 절차:
1) 전원을 켜기 전에 측정할 핀 위치를 먼저 확인합니다.
2) 24V 라인과 3.3V 신호 라인을 시각적으로 구분합니다.
3) 손으로 뾰족한 프로브를 직접 대는 측정 방식을 줄입니다.
4) 후크형 클립 프로브를 먼저 고정한 뒤 전원을 켭니다.
5) 전원공급장치의 전류 제한을 낮게 설정합니다.
6) 측정 중 손이 흔들리지 않도록 리드선을 고정합니다.
7) 측정이 끝나면 전원을 끄고 클립을 분리합니다.
8) 측정 결과와 이상 여부를 기록합니다.

직무 연결:
현장 장비에서도 잘못된 측정은 2차 손상으로 이어질 수 있습니다. 이 경험을 통해 전원·신호 분리, 측정 순서, 전류 제한, 후크형 클립 사용의 중요성을 절차화했습니다.

삽입 자료:
측정 전 체크리스트, 후크형 클립 프로브 예시 사진, 전원 OFF → 클립 고정 → 전원 ON → 측정 → 전원 OFF 순서도, 전원 라인과 신호 라인을 구분한 이미지


10. 테스트 결과와 알람 대응 검증 지표

칠러 CS 직무와 연결되는 지표를 정하고, 실제 테스트 결과를 기록했습니다. 완벽한 성능 수치보다 중요한 것은 기준을 정하고, 측정하고, 로그로 확인하고, 개선 방향을 찾는 과정이라고 판단했습니다.

1차 테스트 조건:
- 실내온도: 약 25°C
- 목표 온도: 27°C
- 테스트 목적: 고온 성능 검증이 아니라, 안전한 저온 조건에서 알람 로직과 복구 조건을 검증하는 것

핵심 확인 지표:
- 목표 온도 도달 시간: 목표 온도까지 걸린 시간. 측정값 100초
- 온도 초과량: 목표 온도보다 더 올라간 온도. 측정값 1.1°C
- 과온 감지 시간: 기준 초과 후 알람 발생까지 걸린 시간. 측정값 5초
- 냉각 후 복구 시간: 팬 동작 후 재가동 가능 온도까지 내려간 시간. 측정값 2초

테스트 케이스:
- TC-01: 목표 온도 도달 및 온도 변화 기록. 결과 Pass
- TC-02: WARNING 감지 후 팬 자동 ON. 결과 Pass
- TC-03: ALARM 감지 후 히터 자동 OFF. 결과 Pass
- TC-04: 온도 미하강 시 RESET 거부. 결과 Pass
- TC-05: 냉각팬 미동작 상황 재현 후 원인 분리. 결과 Pass

해석:
이 테스트는 실제 칠러 성능을 평가한 것이 아니라, 온도 이상 상황에서 알람 발생, 냉각 대응, 히터 차단, RESET 조건 판단이 의도대로 수행되는지 확인한 1차 검증입니다.

향후 보완:
표면온도 센서 추가 후 40°C 이하의 안전 조건에서 반복 측정을 진행하고, 팬 고장, 센서 오류, 히터 고착 시나리오를 추가로 검증할 계획입니다.

삽입 자료:
핵심 지표 숫자 카드, 온도 변화 그래프, 테스트 케이스 Pass 결과 표


11. 현재 구현 범위와 한계

구현한 범위:
- 히터를 이용한 온도 상승 상황 재현
- 목표 온도 대비 현재 온도 비교
- WARNING / ALARM 2단계 알람 구현
- WARNING 상태에서 냉각팬 자동 동작
- ALARM 상태에서 히터 자동 차단
- 온도가 충분히 내려가기 전 RESET 거부
- PC 모니터링 화면 구성
- CSV 로그 저장
- 냉각팬 미동작 고장 재현 및 진단
- 24V 회로 측정 리스크 분석과 재발 방지 절차 정리

한계:
- 실제 칠러의 냉매 사이클을 구현한 것은 아닙니다.
- 압축기, 냉각수 유량, 압력 제어는 포함하지 않았습니다.
- 현재 냉각은 팬 기반이므로 실제 칠러의 냉각 구조와는 차이가 있습니다.
- DHT22는 정밀 표면온도 측정용 센서가 아니므로, 온도 응답 분석에는 한계가 있습니다.

이 한계를 명확히 인식하고 있으며, 프로젝트의 목적은 실제 칠러 제작이 아니라 칠러 CS 직무에서 필요한 알람 대응, 냉각 동작 확인, 안전 복구 판단 흐름을 재현하는 것입니다.


12. 향후 개선 계획

1) 표면온도 센서 추가
히터 표면온도와 챔버 내부 공기 온도를 분리해서 측정하기 위해 DS18B20 또는 K-type 열전대 센서를 추가할 계획입니다.

2) 냉각 성능 검증 강화
팬 동작 전후 온도 변화율, 목표 온도 복귀 시간, 과온 초과량을 반복 측정해 냉각 응답을 더 명확히 확인할 계획입니다.

3) 고장 시나리오 확대
현재는 냉각팬 미동작 상황을 중심으로 검증했습니다. 향후에는 센서 단선, 센서 이상값, 히터 고착, 릴레이 미동작, 팬 전원 불량 시나리오를 추가할 계획입니다.

4) 문서화 강화
알람 매트릭스, 테스트 계획서, 고장 대응 가이드, I/O 목록을 정리해 현장 CS 문서처럼 관리할 계획입니다.

5) 펠티어 냉각 추가 검토
현재는 팬 기반 냉각만 사용하고 있으나, 향후 펠티어를 추가해 능동 냉각 구조로 확장하는 것을 검토하고 있습니다.


13. 이 프로젝트로 보여준 역량

1) 장비 이상 상황을 직접 만들고 검증했습니다.
목표 온도 초과 상황을 만들고, 알람 발생 조건과 냉각 대응 흐름을 직접 확인했습니다.

2) 알람을 단순 표시가 아니라 조치와 연결했습니다.
WARNING에서는 냉각팬을 동작시키고, ALARM에서는 히터를 차단하도록 구성했습니다.

3) 안전 복구 조건을 구현했습니다.
RESET 명령이 들어와도 온도가 충분히 내려가기 전에는 재가동되지 않도록 만들었습니다.

4) 고장 상황을 재현하고 원인을 좁혔습니다.
냉각팬 미동작 상황을 일부러 만들고, 신호, 전원, 부품, 로그를 순서대로 확인해 원인을 분리했습니다.

5) 측정 리스크를 절차로 바꿨습니다.
24V 회로 측정 중 발생한 문제를 분석하고, 후크형 클립 사용, 전류 제한, 전원 OFF 상태에서 측정 지점 고정 등 재발 방지 절차로 정리했습니다.

6) 로그 기반으로 판단했습니다.
감각이나 추측만으로 판단하지 않고, PC 화면과 CSV 로그를 통해 온도 변화와 알람 발생 시점을 확인했습니다.


14. 마무리

이 프로젝트는 실제 칠러 장비를 만든 프로젝트는 아닙니다. 하지만 칠러 CS 엔지니어에게 중요한 목표 온도 이탈 감지, 과온 알람 대응, 냉각 동작 확인, 안전 정지, 복구 조건 판단 흐름을 작은 하드웨어로 직접 구현하고 검증한 프로젝트입니다.

특히 냉각팬 미동작 고장 재현과 24V 측정 리스크 분석을 통해, 문제가 발생했을 때 부품 교체부터 생각하는 것이 아니라 전원, 신호, 센서, 구동부, 로그를 순서대로 확인하는 CS 엔지니어의 진단 습관을 보여주고자 했습니다.

저는 이 경험을 바탕으로 테키스트, GST, 에프에스티의 칠러 CS 엔지니어로서 고객 FAB 현장에서 장비 알람을 정확히 이해하고, 안전하게 대응하며, 재발 방지까지 고민하는 엔지니어로 성장하고자 합니다.
